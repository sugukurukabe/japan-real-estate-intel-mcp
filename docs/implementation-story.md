## 第 1 章 — そもそも何のために作ったか

### 日本の不動産データが「ばらばら」という問題

不動産の意思決定に必要なデータは、じつは公開されています。国土交通省が公示地価、取引価格、路線価を公表し、総務省が人口・世帯データを、国交省がハザードマップを提供しています。

問題は、**これらが別々のウェブサイトやデータ取得口（API）に散らばっていて、一か所で横断的に調べる手段がなかった**ことです。投資家や不動産業者が「名古屋市中区の今後の地価をリスク込みで判断したい」と思っても、複数のサイトを行き来して自分で表計算ソフトに貼り付けるしかありませんでした。

### AI の登場で「聞くだけで答えが出る」世界が近づいた

Claude や ChatGPT のような AI は、自然な日本語で複雑な質問に答えられます。しかし AI は、リアルタイムの地価データや洪水リスクのデータを自分では持っていません。

そこで考えたのが、**AI に「不動産データを取得・分析する専用の道具（ツール）」を持たせる**という仕組みです。この「道具を AI に渡すための標準的な方法」が MCP（Model Context Protocol）です。

MCP を使うと、Claude や ChatGPT に「名古屋市中村区の投資リスクを調べて」と頼むだけで、AI が自動的に複数のデータを取得し、分析して、日本語で答えを返してくれます。

## 第 2 章 — MCP という仕組みを選んだ理由

### MCP とは何か（一行で）

MCP とは、「AI アシスタントに外部の機能やデータを使わせるための共通ルール」です。Claude、ChatGPT、Cursor（開発者向けエディタ）など複数の AI が同じルールに対応しているため、一度作れば複数の AI から使えます。

### なぜ MCP を選んだか

- **標準仕様がある**：Anthropic（Claude の開発元）が公式に定めた仕様なので、AI 側が積極的にサポートしています。
- **ツールを「機能単位」で作れる**：地価分析、リスク評価、レポート生成など、機能を独立したツールとして定義できます。
- **ダッシュボード（画面）も一体化できる**：MCP Apps という拡張機能を使うと、AI のチャット画面の中にインタラクティブな地図やグラフを表示できます。

## 第 3 章 — 全体の設計を決める

### システムの全体像

```
利用者
  ↓（日本語で質問する）
Claude / ChatGPT / Cursor（AI アシスタント）
  ↓（必要なツールを自動的に呼び出す）
Japan Real Estate Intel MCP サーバー
  ├── 33 種類のツール（地価分析・リスク評価・契約支援など）
  ├── 10 都道府県のデータローダー（地価・人口・ハザード・路線価など）
  └── ダッシュボード（地図・グラフ・価格パネルなど）
        ↓（データを取得する）
      17 種類以上のデータソース（公示地価・取引価格・e-Stat・MLIT など）
```

### 3 つの柱

設計を決めるにあたって、3 つの柱を定めました。

1. **ツール**：AI が呼び出せる個別の機能。例：「地価トレンドを予測する」「リスクを評価する」「契約支援パッケージを作る」。
2. **データローダー**：各都道府県のデータを同じ形式で読み込む仕組み。愛知でも東京でも埼玉でも、AI は同じ方法でデータを取得できます。
3. **ダッシュボード**：地図・グラフ・パネルで結果を視覚化するウェブアプリ。AI のチャット画面の中に直接表示されます。

## 第 5 章 — v6.15.0 のハイライト：価格の歪みを見つける

### 3 つの価格を比べると「歪み」が見える

不動産には、同じエリアでも 3 種類の「価格」が存在します。

| 価格の種類 | 説明 |
|---|---|
| **公示地価** | 国が毎年公表する標準的な地価 |
| **路線価** | 税務署が相続税・贈与税の計算に使う地価。公示地価のおよそ 80% が目安 |
| **取引価格** | 実際に売買が成立した価格 |

これら 3 つを組み合わせると、エリアの「過熱度」や「お買い得度」が見えてきます。

- 取引価格が公示地価より大幅に安い → **割安シグナル**（買い場かもしれない）
- 路線価が公示地価の 75% 未満 → **相続税の有利エリア**（相続税を節税しやすい）
- 取引価格が公示地価の 1.3 倍以上 → **市場過熱シグナル**（高値つかみのリスク）

### `detect_arbitrage_signals` ツールの登場

v6.15.0 で新たに追加した `detect_arbitrage_signals`（価格の歪み検出）ツールは、都道府県の全市区町村を自動スキャンして、上記のシグナルを持つエリアを一覧で返します。

Claude に「愛知県で割安シグナルのある市区町村を教えて」と頼むだけで、AI が自動的にこのツールを呼び出し、路線価・公示地価・取引価格を三角測量した結果を返します。

さらにこの分析は、既存の「総合価値スコア」や「Opportunity Radar」にも自動的に組み込まれ、割安エリアには評価ボーナスが加算されるようになりました。

## 第 7 章 — データ基盤の作り方

### 「どの都道府県でも同じ方法で」を実現した仕組み

当初は愛知県だけのコードを書いていましたが、東京・大阪と増えていくにつれ「都道府県が増えるたびにコードを書き直す」のは現実的でないとわかりました。

そこで考えたのが**プレフェクチャー・ローダー（都道府県データ読み込みモジュール）**という仕組みです。

すべての都道府県ローダーは同じ約束事を守ります：
- 「地価データをください」→ 地価データを返す
- 「洪水リスクをください」→ 洪水リスクデータを返す
- 「人流データをください」→ 人流データを返す（または「このデータはありません」と返す）

AI やツールは「どの都道府県か」を気にせず、同じ方法でデータを取得できます。新しい都道府県に対応するときは、その都道府県用のローダーを追加するだけです。

### データの種類

| データ種別 | 出所 | 用途 |
|---|---|---|
| 公示地価 | 国土交通省 不動産情報ライブラリ | 地価分析・予測の基準 |
| 取引価格 | 国土交通省 MLIT API | 実態価格との比較 |
| 路線価 | 国税庁（公示地価から推算） | 税務評価・相続分析 |
| 洪水リスク | 国土交通省ハザードマップ | リスク評価 |
| 地震リスク | 地震調査委員会データ | リスク評価 |
| 人口統計 | 総務省 e-Stat（国勢調査） | 将来性分析 |
| 人流データ | 民間データ（NTT ドコモ等参考） | 商業・出店分析 |
| 教育データ | 各都道府県公開データ | ファミリー評価 |
| 企業立地 | 経済センサス | 法人需要分析 |
| 交通・商業・医療 | 各都道府県公開データ | 総合スコア |
| 用途地域 | 国土交通省 | 容積率・建蔽率 |
| 空き家率 | 住宅・土地統計調査 | エリアリスク評価 |
| 人口推計 | 社会保障・人口問題研究所 | 将来性（2050 年まで） |
| 町丁目データ | 名古屋市オープンデータ | 超ローカル分析 |
| PLATEAU 3D | 国土交通省（3D 都市モデル公開プロジェクト） | 建物形状・日照分析 |

## 第 8 章 — UI（ダッシュボード）の作り方

### 「チャットで聞いて、地図で見る」体験

このシステムの使い方は大きく 3 通りあります。いずれの AI でも、地図やグラフを含むダッシュボードが会話画面の中に直接表示されます。

**パターン 1：Claude / Cursor — チャット + 埋め込みダッシュボード**
Claude Desktop や Cursor に自然な日本語で質問すると、AI が自動的にツールを呼び出します。MCP Apps 対応により**ダッシュボードが AI チャット画面の中に直接表示**され、地図操作・ボタンクリックが AI に自動伝達されます。

**パターン 2：ChatGPT — チャット + 埋め込みダッシュボード**
2026年1月より ChatGPT も MCP Apps に対応し、Claude / Cursor と同様に**インタラクティブな地図・グラフ・価格パネルが会話内に直接表示**されます。ツール呼び出しだけでなく、ダッシュボード内のボタン操作も AI に自動的に伝わります。

**パターン 3：ブラウザで直接開く（AI なしでも使える）**
`https://realestate-mcp.jp/dashboard.html` をブラウザで開くと、地図を操作しながらデータを見られます。iPad に対応しており、現地調査でも使えます。

### ダッシュボードに組み込んだ機能

- **2D 地図**（Leaflet）：都道府県を選択すると地価ヒートマップが表示。市区町村をクリックすると詳細パネルが開く
- **3D ビュー**（Three.js）：建物の立体表示と日照シミュレーション
- **インサイトパネル**：地価・リスク・人流・総合スコア・価格トライアングルを右側に表示
- **Opportunity Radar**：全市区町村のスキャン結果をカードで表示
- **価格トライアングルパネル**：路線価・公示地価・取引価格を横棒グラフで比較

## 第 9 章 — 品質をどう担保したか

### テストと型チェック

ソフトウェアの「見えないバグ」を防ぐために、自動テストを徹底しました。

- **696 のテスト**：データ読み込み、ツールの動作、スキーマ検証、HTTP 通信、UI の表示など、あらゆる箇所を自動でチェック
- **42 のテストファイル**：機能ごとにテストを整理
- **TypeScript 型チェック**：コードを書く段階で「型の矛盾」がないか自動でチェック（ゼロエラー）
- **CI（継続的インテグレーション）**：コードを更新するたびに GitHub 上で自動テストが走る仕組み

### セキュリティ

- **セキュリティヘッダー**：Web サーバーへの不正アクセスを防ぐ設定を自動適用
- **レートリミット**：短時間に大量のリクエストが来ても対処できる流量制限
- **CodeQL**：週次でプログラムのセキュリティ上の弱点を自動スキャン
- **最小権限でのサーバー運用**：Docker（コンテナ技術）で本番環境を安全に隔離
- **OAuth 2.1 認証**：業界標準の認証プロトコルで本番アクセスを管理
